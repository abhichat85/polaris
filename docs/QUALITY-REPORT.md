# Polaris Quality Report

> **Two-layer eval system as of D-048:**
> - **v1 (process):** `pnpm test:eval` — fast, deterministic, mock LLM. Measures whether the agent uses the right tool sequences (read-before-edit, edit_file vs write_file, layer-2 error recovery). Sub-second; runs on every commit.
> - **v2 (output):** `pnpm test:eval:real` — slow, real Anthropic API + real Playwright. Measures whether the generated APP actually works: boots, no console errors, click flows, persistence round-trips, visual diff against baseline. Nightly + on-demand via workflow_dispatch.
>
> A change to the agent loop must keep v1 green AND not regress v2 pass rate. The v1 measures *agent process*, the v2 measures *agent output*.

## What v1 measures

These scenarios verify the agent's *decision making* with a deterministic LLM stub.
They prove the loop, executor, sink, and tools wire together correctly under realistic
agent behavior patterns. They do NOT measure:

- Real LLM output quality (see v2 below)
- Live E2B sandbox boot (needs E2B_API_KEY)
- Browser-side preview rendering (covered by v2)
- End-to-end Stripe webhook flow (needs Stripe test events)

See `docs/MANUAL-TEST-PLAN.md` for the manual checklist + `scripts/eval-live.ts`
for the legacy live-API runner.

## v2 (real-eval) — output quality

> Generated by `pnpm test:eval:real`. Boots the generated app via Playwright and asserts visual + behavioral correctness. Run nightly + on-demand. Costs ~$3–8 per scenario × 8 = up to ~$64 per nightly.

### Scenarios

| ID | Title | What it tests |
|---|---|---|
| 01 | Static marketing page | Multi-section page renders, responsive at 3 breakpoints, no console errors |
| 02 | Auth flow with Clerk | /sign-in + /sign-up render Clerk form, /dashboard exists |
| 03 | Product list with cart | 6 product cards, add-to-cart increments header counter, persists across reload |
| 04 | Contact form validation | Empty submit shows 3 errors, invalid email rejected, valid input → success state |
| 05 | Dark/light toggle | Header toggle flips html class, preference persists in localStorage |
| 06 | Fix runtime bug | Pre-seeded TypeError-on-click — agent diagnoses + fixes via D-045/D-046 path |
| 07 | Image-to-UI | Reference invoice screenshot → agent builds matching layout |
| 08 | Fullstack todo | Convex schema + mutations + queries; add/toggle/delete; reload-persistent |

### How to run locally

```bash
pnpm install
pnpm add -D pixelmatch pngjs            # eval extras for visual diff
pnpm exec playwright install chromium
ANTHROPIC_API_KEY=... pnpm test:eval:real
```

### Filtering + baseline updates

```bash
EVAL_SCENARIO_IDS=03-product-list-cart,06-fix-runtime-bug pnpm test:eval:real
EVAL_UPDATE_BASELINES=true pnpm test:eval:real    # rewrite all baselines
```

### When v2 fails

Visual-diff mismatch → look at the `.diff.png` next to the candidate. Genuine intent (design system update) → regenerate baselines and review the diff in PR. Regression → fix the agent loop or scenario before merging.

Behavioral assertion fail → read the assertion's console-message capture in the report; usually the generated app threw a runtime error the agent didn't catch (and v1 wouldn't have caught either, which is exactly why v2 exists).
